# MonkeyBench — 研究与设计方案(RDP / Research & Design Plan)

> 本文是项目的研究设计方案,等价于一份"非产品版 PRD"。配套文档:`CLAUDE.md`(给 Claude Code 的开发上下文)。已有代码骨架见末尾 §12。

> **一句话**:让多套策略和一群随机"猴子"在同一套虚拟撮合规则下同台竞技,用排行榜回答"机器学习在股市里到底有没有用"。

---

## 0. 这是什么 / 不是什么

- **是**:个人量化**研究、验证、兴趣**工具。可小范围分享给朋友共同验证。
- **不是**:SaaS、不是产品、不接券商、不碰真钱。所有"订单 / 账户 / 盈亏"都是**虚拟核算**。
- **性质决定取舍**:工程化程度以"自己和朋友能跑、能相信结果"为准。不做多租户、计费、SLA、账号体系这些产品化的东西。

---

## 1. 研究问题(为什么做)

- **核心问题**:ML(尤其是截面排序模型)在**扣掉真实交易成本**后,能否稳定跑赢"运气基线"?
- **运气基线** = 一群随机交易猴子里最好的那只 + 买入持有。这是内置的**零假设**。
- **次要问题**:不同市场规则(T+1、涨跌停分层)对策略表现的影响;数据质量对回测可信度的影响。

---

## 2. 非目标(明确不做)

- ❌ 不接实盘 / 券商 API,不下真实订单,不处理任何资金。
- ❌ 不做 SaaS / 多租户 / 计费。
- ❌ 不追求低延迟 / 高频。**日频为主**。
- ❌ 当前不做美股、北交所(后续可选加)。

## 3. 范围(Scope)

### 3.1 市场分期

| 阶段 | 市场 | 数据来源 | 状态 |
|---|---|---|---|
| **Phase 1(当前)** | 沪深 A 股:主板、创业板、科创板、ST/*ST | Tushare 优先 + 免费源 fallback | 重点 |
| Phase 2(看情况) | 港股 | **仅限 akshare 能提供的** | 待定 |
| 暂不做 | 美股、北交所 | — | 后续 |

### 3.2 A 股市场规则(Claude Code 实现依据)

| 板块 / 类型 | 涨跌停 | 最小买入单位 | 备注 |
|---|---|---|---|
| 主板(沪 60 / 深 000,001,002) | ±10% | 100 股整数倍 | T+1 |
| 创业板(300/301) | ±20% | 100 股整数倍 | T+1 |
| 科创板(688) | ±20% | **≥200 股,之上 +1 股递增** | 卖出可不足整手;T+1 |
| ST / *ST | ±5% | 100 股整数倍 | T+1 |

- **T+1**:全 A 股当日买入不可当日卖出(骨架已实现 `lock_same_day_buy`)。
- **新股特例**(Phase 1.5 细化,先按常态处理):科创板 / 创业板上市前 5 个交易日无涨跌停;主板首日临停机制。
- **成本(近似值,落地须核实当前规则)**:
  - 佣金:约万 2.5,最低 5 元,**双边**
  - 印花税:**0.05%,仅卖出**(2023-08 减半后)
  - 过户费:约十万分之一(0.001%),双边
  - 滑点:按模型假设(小盘 / 封板更大)

> ⚠️ 涨跌停幅度、新股规则、各项费率都随监管变动。落地时以**交易所 / 中证登当前规则**为准,不要照抄本表。

## 4. 系统架构(三层)

```
┌──────────────────────────────────────────────────────────┐
│  研究层(离线,Qlib / vectorbt)                            │
│  因子工程 + ML 训练 + 截面排序                              │
│  ── 产出 ──> 信号文件 (date × symbol → 分数 / 目标仓位)     │
└───────────────────────────┬──────────────────────────────┘
                            ▼ (信号文件 = 研究层与执行层的唯一接口)
┌──────────────────────────────────────────────────────────┐
│  执行层 / 竞技场(唯一引擎,已有骨架 src/arena/*)          │
│  策略(规则型 或 读信号型)→ 套市场规则+成本 → 撮合        │
│  喂历史数据 = 回测   |   喂每日拉盘 = 纸面实盘(同一引擎)   │
└───────────────────────────┬──────────────────────────────┘
                            ▼
┌──────────────────────────────────────────────────────────┐
│  排行榜 / 归因层                                            │
│  吃 append-only 事件日志 → 指标 + 排名(含猴子基准)        │
└──────────────────────────────────────────────────────────┘
```

**铁律**:排行榜上的官方数字**只能来自竞技场引擎**。Qlib / backtrader 自带回测只用于研究层 sanity check,其数字不得直接上榜——否则不同成交假设之间是"关公战秦琼",排行榜失去意义。

## 5. 数据层(本项目的主战场)

### 5.1 多数据源 + 优先级

- **主源**:Tushare Pro(高级 API,已有)。**接入方式待定**:仓库已有 `TuShareMCP`(MCP 服务器配置)与 `.env`(`TUSHARE_API_KEY`),可走 MCP 或 `tushare` SDK,M1 前二选一(见 §11)。
- **Fallback**:akshare、baostock(免费)。
- **SourceRouter**:按 `(symbol, field, 时间段)` 依优先级取数,失败逐级降级,并记录 **provenance**(每条数据来自哪个源、哪个版本)。
- **本地缓存**:列式存储(parquet / DuckDB)+ 缓存 manifest。**先查缓存,miss 才打 API**。

### 5.2 数据质量(不可妥协)

- **复权**:前 / 后复权(qfq/hfq)。收益必须用复权价算,否则除权除息当天会凭空出现巨大涨跌。
- **幸存者偏差**:回测**必须含已退市股票**,否则只在赢家里测,结果虚高。
- **Point-in-time**:基本面因子只用"当时已知"的版本,后修正版本是隐性未来函数。
- **停牌**:正确处理停牌段(不可当成正常交易日)。

### 5.3 历史数据导入工具(Backfill ETL)

**目标**:在 API 合规前提下,用一段时间把 **≥10 年**历史全部补齐。

**设计**:
- 枚举全市场 universe(**含退市股**)× 日期区间。
- **限速**(token bucket),尊重 Tushare 积分 / 每分钟调用上限,以及各源 ToS。
- **可断点续传**:checkpoint manifest 记录"已拉 / 未拉 / 失败待重试"。
- **增量更新**:每日跑增量,补齐当日。
- 落地到本地缓存,带 provenance 与数据版本号。

> Tushare 即便高级版也有每分钟调用与积分限制,backfill 必须**慢**、必须**可续**。不要写成一把梭的脚本。

## 7. 竞技场与猴子(方法论核心)

- **20+ 随机交易猴子**(不同种子)= 噪声地板 / 运气天花板。最好那只猴子的夏普,就是这个账号规模下的运气上限。
- **参照基准**:买入持有(沪深 300 等)、等权、定投。
- **判决规则**:任何精心设计的策略,扣完成本后干不过最好的猴子、也干不过买入持有 → 判定**没有真 edge**。
- 这等于给排行榜**内置了一个零假设分布**,白送的显著性检验。(骨架的 `summarize()` 已自动给出此判决。)

## 8. ML 框架

- **任务定义**:**截面排序**(在一篮子股票里排谁跑赢谁),而非"明天涨跌"二分类(信噪比太低,扣成本必败);或 **meta-labeling**(简单规则定方向,ML 只决定下不下注 + 仓位多大)。
- **防过拟合**:
  - train / validation / **holdout** 三段时序切分(绝不随机切)。
  - **walk-forward** 滚动(用"当时的过去"预测"当时的未来")。
  - holdout 严格少看——每看一次消耗一次统计信用。
  - CV 用 **purged + embargoed**(训练 / 测试间留隔离带,防标签窗口重叠泄漏)。
- **过拟合度量**:用 **Deflated Sharpe Ratio (DSR)** / **Probability of Backtest Overfitting (PBO)**,而不是拍脑袋公式。况且猴子已给了你经验版零假设,可直接对比。
- **模型**:Qlib Model Zoo —— LightGBM/XGBoost 做基线,往上 TRA / HIST / Transformer;**概念漂移 / 风格切换**用 DDG-DA / ADARNN。
- **可选**:Qlib RD-Agent 用 LLM 自动挖因子(其"少 70% 因子、2 倍收益"是宣传口径,**待验证**)。

## 8.5 公司多元分析 + LLM 报告(综合评分 → 报告)

> 把"综合评分"从"喂竞技场的信号"扩展为**给人看的公司诊断报告**。用既定的
> **LLM agent + skill 工具链(非 NL2SQL)**:LLM 只编排调用确定性工具 + 解读撰写,
> **数字一律来自工具,不让 LLM 自己算/编**。

- **目标**:给定一家(或一组)A 股公司 → 拉全维度数据 → 算成熟综合指标 + 同业对标 → LLM 生成结构化分析报告。
- **输入**:`ts_code`;可选 对标范围(默认同行业)、as-of 日期(point-in-time)。
- **数据维度**(Tushare 权限内,从 DuckDB 读):基础(行业/市值/ST)、行情(动量/波动/回撤)、**财务(PIT,用 ann_date)**、估值(PE/PB/PS/股息)、特色(概念/资金流/筹码/龙虎榜/融券/量化因子)。
- **综合指标层(确定性,研究层)**:
  - **Piotroski F-Score**(0–9 财务健康,等权)、**AQR QMJ**(盈利+成长+安全+分红,四维 rank 加权)、**Greenblatt Magic Formula**(ROC+EY 排名)、**Altman Z-Score**(破产距离)、自定义综合质量分。
  - **同业对标**:各指标在**行业内百分位排名**(对标核心 + 行业中性化)。
- **LLM 报告层(agent + skill)**:
  - 确定性工具(纯 Python,内部查 DuckDB):`company_profile` / `financials`(PIT) / `composite_scores` / `peer_percentile` / `price_action` / `special_data`。
  - agent(OpenAI 兼容,DeepSeek 等)tool-use 循环编排 → 解读 → 撰写;**工具核心一份,多协议暴露**(function-calling + 可选 MCP)。
  - 报告结构:公司概览 → 财务健康(各分 + 解读) → 同业对标(百分位) → 估值 → 风险点 → 综合结论。
- **铁律呼应**:LLM 不碰 SQL、数字全来自工具;报告标注 as-of(回溯报告不用未来数据);**非投资建议**(含免责)。
- **与架构关系**:复用数据层 + 研究层(综合分也可喂 `CompositeScoreStrategy` 上竞技场) + 洞见层;Web `/app` 里一个页面:输入公司 → 报告 + 评分雷达图。

## 9. 技术栈(定稿,详见 CLAUDE.md「技术栈」)

- **语言 / 包管理**:Python ≥ 3.13,**uv**。引擎核心**零依赖**。
- **数据库**:**DuckDB**(行情/因子/回测产出,OLAP)+ **SQLite/WAL**(应用状态,经 SQLAlchemy)。**不上 PG**(单实例自用)。
- **数据源**:`tushare`(主,SDK)/ `akshare` / `baostock`(fallback);缓存 DuckDB + parquet。
- **后端 / 前端**:FastAPI + **HTMX + Tailwind**(服务端渲染;K线 = lightweight-charts JS 孤岛)。
- **LLM 洞见**:agent + skill 工具链(**非 NL2SQL**),OpenAI 兼容 API 可换模型(DeepSeek V4 / GLM-5.2 / MiniMax-3)。
- **研究**:`pyqlib`;`vectorbt`(参数快筛,可选)。
- **调度**:`APScheduler`(每日增量 + 纸面实盘)。
- **部署**:exe.dev 单 VPS + docker compose(TLS/分享平台负责 + app 层简单 auth)。
